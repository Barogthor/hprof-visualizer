# Tech Spec Review: Refactor first_pass.rs into focused modules

**Reviewed:** 2026-03-09
**Spec:** `docs/implementation-artifacts/tech-spec-refactor-first-pass-modules.md`
**Source verified against:** `crates/hprof-parser/src/indexer/first_pass.rs` (2849 lines)

---

## Findings

1. **[Critical] Test counts are wildly wrong — 61 actual vs 46 claimed.** The spec claims 29 tests in `mod tests` and 17 in `mod builder_tests`, totaling 46. The actual file has 37 tests in `mod tests` and 24 in `mod builder_tests`, totaling 61. This error propagates to AC-1 ("all 46 existing tests pass") and the Testing Strategy section ("All 46 existing tests"). Every acceptance gate that references "46" is wrong and would give false confidence that all tests passed when 15 are silently missing.

2. **[Critical] Task ordering makes Task 3 break compilation.** Task 3 says to delete `extract_heap_object_ids` and replace it with the unified `extract_heap_segment` + `merge_segment_result`. But `extract_heap_object_ids` is called from two places in `run_first_pass`: (a) inline during the scan loop (line 213) and (b) the sequential fallback after the loop (line 546). Task 3's instructions address case (a) via "replace the inline call" but the `merge_segment_result` function signature in Task 3 takes individual `&mut` params, while the `FirstPassContext` struct is not introduced until Task 6. This means Task 3 must define a temporary `merge_segment_result` signature taking 6+ mutable references, then Task 6 rewrites it to take `&mut FirstPassContext`. The spec never acknowledges this double-refactor or the intermediate throw-away signatures.

3. **[High] `push_warning` and `push_suppressed_summary` are not assigned to any module in Tasks 2-5.** The spec mentions they "become methods on `FirstPassContext`" in ADR-1 and Task 6, but Tasks 2-5 execute first. During Tasks 3-5, these free functions are still needed by `record_scan.rs`, `heap_extraction.rs`, and `mod.rs`. The spec never says which module they live in during the intermediate steps (Tasks 2-5), and never explicitly moves them. They presumably stay in `mod.rs` until Task 6 absorbs them, but this is never stated.

4. **[High] `#[cfg(test)]` production code line count is inflated.** The spec claims "~380 lines of `#[cfg(test)]`-only code in the production file." The actual count of `#[cfg(test)]`-annotated items outside the test modules is approximately 203 lines (SUB_DIVIDE_THRESHOLD: 2L, skip_heap_object: 55L, extract_class_dumps_only: 30L, PrepassResult: 5L, prepass_and_subdivide_segment: 61L, subdivide_segment: 50L). Overstating by ~85% undermines the problem statement's credibility.

5. **[High] Benchmark file completely unmentioned.** `crates/hprof-parser/benches/first_pass.rs` imports `hprof_parser::indexer::first_pass::run_first_pass`. While this import path should survive the file-to-directory module conversion (Rust resolves both), the spec never mentions the benchmark file in "Files to Reference," "files_to_modify," or the Testing Strategy. The benchmark file should be listed as a verification target ("run `cargo bench` to confirm benchmark still compiles").

6. **[High] `builder_tests` feature gate is misdescribed.** The spec says `builder_tests` is "feature-gated on `test-utils`" (frontmatter line 24). The actual gate is `#[cfg(all(test, feature = "test-utils"))]` — a conjunctive gate requiring both `test` AND `test-utils`. If the move to `tests.rs` (which is already `#[cfg(test)]` via `mod tests;`) preserves the `#[cfg(all(test, feature = "test-utils"))]` annotation, it would be redundant but harmless. If someone simplifies it to just `#[cfg(feature = "test-utils")]`, the tests would compile in non-test builds. The spec should explicitly state whether the inner gate should be simplified to `#[cfg(feature = "test-utils")]` or kept as-is.

7. **[Medium] `parse_and_insert` signature in Task 5 is under-specified and likely unworkable.** The proposed signature `parse_and_insert<T>(tag, payload_start, header_length, payload_cursor, parse_fn, insert_fn, warnings, suppressed) -> bool` does not account for the fact that each record type's parse function has a different signature. `parse_string_ref` takes `(cursor, id_size, header_length, body_start)` — four params. `parse_load_class` takes `(cursor, id_size)` — two params. `parse_stack_frame` takes `(cursor, id_size)`. The `LoadClass` insert closure also needs access to `result.index.strings` and `data` for class name resolution, making it capture a different set of state than the other closures. A single generic function may not cleanly abstract all 5 arms without awkward closure gymnasics or becoming more complex than the code it replaces.

8. **[Medium] `StringInUtf8` parse call uses `body_start` (line 266-268), which `parse_and_insert` does not model.** The `StringInUtf8` arm passes `header.length` and `body_start` (computed as `payload_start as u64`) to `parse_string_ref`. No other record type passes these extra arguments. The proposed `parse_and_insert` generic helper would need special handling for this one case, defeating the purpose of unification, or the `parse_fn` closure would need to capture these values from the enclosing scope, making the abstraction leaky.

9. **[Medium] `std::collections::HashSet` dependency in `extract_obj_refs` is not mentioned anywhere.** The function uses `std::collections::HashSet` inline (fully qualified path, line 1418) to detect class hierarchy cycles. When this function moves to `thread_resolution.rs`, it will still compile (fully qualified paths work), but the spec's dependency analysis for `thread_resolution.rs` only mentions `hprof_primitives::value_byte_size`. Missing dependencies in the spec create blind spots during implementation.

10. **[Medium] Task 6 introduces `FirstPassContext` AFTER Tasks 3-5 already defined module boundaries, forcing a second rewrite of every module's public API.** Tasks 3-5 define `extract_all(...)`, `resolve_all(...)`, and `scan_records(...)` with individual parameter lists. Task 6 then rewrites all three signatures to accept `&mut FirstPassContext`. This is not a "final wiring task" — it is a destructive rewrite of every interface established in the preceding three tasks. A more efficient ordering would introduce `FirstPassContext` first (as Task 2), then extract modules that already use the context struct, avoiding throwaway intermediate APIs.

11. **[Medium] AC-8 ("no `#[cfg(test)]` in production code") conflicts with the `#[cfg(test)] mod tests;` declaration that must remain in `mod.rs`.** The acceptance criterion says "Given the production code (excluding `tests.rs`), when searching for `#[cfg(test)]`, then no matches are found." But `mod.rs` is production code and must contain `#[cfg(test)] mod tests;` per Task 7. The AC needs to either exclude `mod tests;` declarations or be reworded to say "no `#[cfg(test)]` annotations on functions, structs, or constants."

12. **[Low] `PARALLEL_THRESHOLD` visibility change from `const` to `pub(super)` is a semantic shift the spec does not justify.** The constant is currently module-private (`const`). Task 2 promotes it to `pub(super)` so that `heap_extraction.rs` and possibly `record_scan.rs` (which contains the `defer_heap_extraction` heuristic) can access it. This is correct mechanically, but the spec never explains which sub-modules need it or why, making it look like an accidental over-exposure.

13. **[Low] The spec claims `run_first_pass` is "535 L" but it is actually 539 lines (lines 128-663, accounting for the closing brace before `skip_n`).** Minor, but sloppy measurement in a spec whose entire justification is "the function is too big" does not inspire confidence.

14. **[Low] No rollback plan or intermediate validation beyond "cargo test must pass."** For a 7-task refactor of a 2849-line file, the spec should recommend tagging or branching after each task so that a failed Task 5 does not require unwinding Tasks 2-4 by hand. The instruction "never proceed with failing tests" is necessary but not sufficient — it does not address how to recover if a task leaves tests passing but introduces a subtle semantic change (e.g., warning message text differences from the unification in Task 3).

15. **[Low] The spec's "Scope / Out of Scope" section says "No logic or behavior changes" and "No performance optimizations," but Task 3 explicitly deletes `extract_heap_object_ids` and unifies it with `extract_heap_segment_parallel`.** The unified path uses a local `HeapSegmentResult` + merge for the sequential case, introducing a temporary `Vec` allocation that did not exist before. The spec itself acknowledges this ("temporary Vec allocation in sequential path is negligible vs disk I/O"). Calling this "no behavior change" is technically defensible but intellectually dishonest — it changes the allocation pattern, the warning text (parallel path uses `result.warnings.push` directly, sequential path uses `push_warning` with suppression), and the progress reporting granularity (sequential path reported progress per sub-record, the unified path would report per segment).
