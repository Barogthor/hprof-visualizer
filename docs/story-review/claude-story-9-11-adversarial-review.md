# Adversarial Review — Story 9.11: Go-to-Pin Async Navigation & Scroll-to-Target

**Date:** 2026-03-14
**Reviewer:** Claude (adversarial)
**Content type:** Story specification

---

## Findings

1. **`scroll_to_cursor` offset clamp allows viewport past end of list.** The spec's code snippet clamps offset to `items.len().saturating_sub(1)`, but the correct upper bound is `items.len().saturating_sub(visible_height)`. With 10 items and `visible_height=8`, setting offset to 9 means the viewport shows items 9..17 — 7 blank rows. This is a logic bug baked into the spec that will ship if implemented verbatim.

2. **In-frame continuation cap contradicts itself (10 vs 3).** Task 1.5 says "Cap at 10 consecutive in-frame steps then yield." The Guardrails section says "Cap at 10 (or 3 if `flat_items().len() > 5000`)." The conditional threshold appears nowhere in the task list or test matrix. Test 5.6 only tests "after 10 cached steps" — the 5000-item branch has zero coverage and will be silently omitted during implementation.

3. **No AC for "any user input cancels navigation" (RT-A3).** Task 4.4 introduces a major UX behavior — any keypress except Escape implicitly cancels a multi-minute walk. This is user-visible and should be governed by an AC with explicit expected behavior. Test 5.10 only covers arrow keys; mouse scroll, resize events, and other non-keyboard events are unspecified.

4. **"Any input cancels" is actively hostile on 70 GB dumps.** The primary motivation for this story is large collections on massive dumps where walks take minutes. An accidental keystroke (cat on keyboard, bumped touchpad, terminal resize) kills the entire navigation. The user must re-trigger `g` and wait again from scratch. This undermines the core use case. The spec should either whitelist safe inputs (resize, mouse events) or restrict cancellation to deliberate Escape only.

5. **Missing AC for stale context / walk restart.** The Guardrails section describes a stale-context scenario where an evicted parent triggers a full walk restart with "Pin context changed, retrying..." in the status bar. This is user-visible behavior with no corresponding AC. Without an AC, it's untestable. Test 5.7 tests "stale context detection" but the restart behavior is vaguely defined — restart from scratch or from last valid step?

6. **`AwaitedResource` enum cannot represent the stale-context restart.** The enum has two variants: `ObjectExpansion(u64)` and `CollectionPage(u64)`. When stale context is detected on retry, the walk must restart. But there's no variant to signal "restarting" and no field for the original full path (since `remaining_path` only holds the tail). The struct needs either a `FullRestart` variant or a separate `original_path` field — neither is designed.

7. **`open_stack_for_selected_thread()` destroys expansion state — thread switch is broken.** At `mod.rs:182-188`, `open_stack_for_selected_thread` creates a brand new `StackState` every time. This wipes all expansion phases, collection chunks, and field row maps of the previous thread. If go-to-pin targets a different thread than the current one, the walk must expand everything from scratch after the switch. But critically: if the walk was mid-progress on the old thread and the user cancels, they've lost all expansion state on that thread too. The spec (RT-A5) punts this as "pre-existing limitation" but it's a data loss bug for the feature's core path.

8. **Polls run inside `render()` — not before input handling.** The current code calls `poll_expansions()` and `poll_pages()` at the top of `render()` (`mod.rs:1752-1753`). The spec's guardrail (FM6) requires "input handling → poll_expansions → poll_pages" order. But in `run_loop` (`mod.rs:1998-2013`), the sequence is: `render` (which polls) → `event::poll` → `handle_input`. This means polls run BEFORE input in every frame. Refactoring this ordering is a prerequisite that no task addresses. Moving poll out of render into the main loop is a non-trivial structural change — it needs its own subtask.

9. **Spinner will not animate under current event loop.** The event loop (`mod.rs:2001`) uses `event::poll(16ms)`. When no keys are pressed, `event::poll` returns `false` and the loop jumps back to `terminal.draw()` — so far so good, the spinner animates at 60fps. BUT: the `if event::poll(...)` is a conditional with `&&` — if poll returns `false`, the `event::read()` is skipped and the loop goes back to render. This actually works. However, task 3.3 says "increment `spinner_tick` each frame" — there's no explicit task to add the increment call inside `render()` or `update()`. The increment location is unspecified, just the mechanism.

10. **`pending_navigation` retry in `poll_pages` restarts the FULL walk.** At `mod.rs:1598-1612`, when the awaited collection loads, the retry calls `navigate_to_path(thread_id, &nav_path)` with the **full original path**, not just the remaining segments. This re-walks already-resolved segments (thread switch, frame expansion, etc.). The spec says "Each resolved step is dropped from `remaining_path` — the walk never restarts from scratch" (Design Decision #1), but the current retry code and the proposed `PendingNavigation.remaining_path` design are incompatible. Task 1.6 says "resume the walk with remaining segments" but the current code passes the full path. This gap is not called out and will cause confusion.

11. **AC2 progressive rendering has no test coverage.** AC2 is the headline UX improvement: "the view re-renders before continuing to the next step" and "the user sees the navigation unfold." None of the 11 tests verify intermediate rendering between walk steps. The test suite only checks final state. Progressive rendering is architecturally complex (yield between steps, re-render, resume) and has zero validation.

12. **Empty `remaining_path` after resume is unhandled.** When the last segment of the walk is the deferred one (e.g., the final `CollectionEntry` is the one that triggers async page load), the resume will have `remaining_path = []`. If `poll_pages` resumes with an empty remaining path, the walk function receives an empty slice. The spec's `navigate_to_path` refactor (Task 1.2) accepts `&[PathSegment]` but lines 358-360 of the current code return `PartialAt(FrameId(0))` for empty segments — a silent wrong-position bug. No test covers this edge case.

13. **`navigate_to_path` retry uses wrong `thread_id` on thread switch.** At `mod.rs:1607-1611`, the retry path reads `thread_id` from `thread_list.selected_serial()` — the currently selected thread. But if the user manually switches threads after a nav deferral (before the page loads), the retry will navigate within the wrong thread. The `PendingNavigation` struct stores `thread_id`, but the current retry code doesn't use it. Task 1.6 must explicitly use `pending_navigation.thread_id`, but this existing bug in the retry path is not called out as something to fix.

14. **No task for refactoring `poll_expansions` to resume walks.** Task 1.6 says "Update `poll_pages()` and `poll_expansions()` to check `pending_navigation.is_some()` and resume the walk." But `poll_expansions` currently (`mod.rs:1685+`) does NOT check `pending_navigation` at all. It only applies expansion results to `stack_state`. Adding walk resumption to `poll_expansions` requires: checking if the completed expansion matches `AwaitedResource::ObjectExpansion(oid)`, extracting remaining path, calling the walk function, and handling the result. This is substantial new logic with no dedicated subtask — it's buried in a single line of task 1.6.

15. **Test 5.11 integration test assumes `HprofTestBuilder` supports pinning.** The test says "pin an item, navigate away, press `g`, verify cursor ends on the correct row with correct viewport offset." But `HprofTestBuilder` is a test fixture for the parser/engine layer. Pinning is a TUI-layer concept (`PinnedItem` in `App`). The test must construct an `App`, manually insert a `PinnedItem`, simulate input events, and inspect state. This is more complex than a simple builder call and the test description is misleading about the setup required.
