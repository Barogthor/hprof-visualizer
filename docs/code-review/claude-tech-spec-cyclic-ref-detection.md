# Tech-Spec Review: Cyclic Reference Detection

**Reviewed:** 2026-03-09
**Spec:** `docs/implementation-artifacts/tech-spec-cyclic-ref-detection.md`
**Verdict:** Multiple issues ranging from incorrect line references to missing edge cases and an incomplete task list.

---

## Issues Found

1. **`flat_index` does NOT extract `frame_idx` -- the spec's claim about line 666 is fabricated.** The spec (Task 2, third bullet) says `flat_index` at "line 666" extracts `frame_idx` from cursor variants and needs an `OnCyclicNode` arm. In reality, `flat_index` (line 474) is a trivial 3-line function that calls `flat_items()` and does `position()` -- it has zero match arms and no `frame_idx` extraction. The spec confuses it with the `build_items` frame-selection check around line 662-667, which is a `matches!` guard inside the rendering loop, not `flat_index`. This is a factual error that will confuse any developer following the task list.

2. **Every line number reference is fragile and several are already wrong.** The spec cites specific line numbers like 158, 432, 511, 552, 666, 721, 739, 837. These are snapshot values that drift with any edit. Worse, some are already inaccurate: `emit_object_children` starts at line 511 (spec says 511 -- OK), but the `build_object_items` call site "line ~721" is actually line 721 in `build_items`, not a call site in isolation. The spec should reference function names and structural landmarks, not fragile line numbers.

3. **Missing Task: the `build_items` frame-selection `matches!` macro (line 662-667) also needs `OnCyclicNode`.** The spec lists three sites in Task 2 for match-arm updates but omits the `is_selected` check in `build_items` (lines 662-667) which groups `OnVar`, `OnObjectField`, and `OnObjectLoadingNode` to highlight the parent frame row when any child cursor is active. If `OnCyclicNode` is not added here, the frame row will lose its highlight when the cursor sits on a cyclic marker. This is a visual regression the spec completely misses.

4. **No test for keyboard navigation (move_up / move_down) across cyclic nodes.** `move_up` and `move_down` (lines 451-470) use `flat_items()` and search by `PartialEq`. The spec assumes `OnCyclicNode` will "just work" with navigation because it appears in `flat_items`, but there is no test verifying that up/down arrow keys traverse cyclic nodes smoothly. If the `PartialEq` derive on the new variant has issues or `flat_items` emits items in a surprising order near cycle boundaries, navigation will break silently.

5. **The `&mut HashSet<u64>` parameter is unnecessary -- `&HashSet<u64>` suffices for `build_object_items`.** Task 4 says to pass `&mut HashSet<u64>` to `build_object_items`, but that function is `&self` (immutable borrow on `StackState`). Passing `&mut HashSet` works syntactically, but the spec doesn't explain WHY mutable is needed for the rendering path. For `emit_object_children` mutability makes sense (insert/remove for DFS backtracking), but `build_object_items` could use the same pattern or an immutable snapshot. The spec doesn't consider whether both functions even need backtracking -- if expansion_state is global per object_id and you only care about ancestors, you need mutability in both, but the spec never justifies this for the render path. An implementer might reasonably use `&HashSet` and get stuck.

6. **Backtracking (remove after iteration) is incorrect when the same object_id appears at multiple depths in a non-cyclic path.** The spec says "insert before iterating, remove after -- standard DFS backtracking." But if object A has two fields both pointing to object B (not a cycle, just shared reference), the first field's recursion inserts B, processes it, removes B. The second field's recursion inserts B again -- fine. However, if object B points to object C which points back to B, the first branch detects the cycle, but after backtracking removes B, the second branch will NOT detect B as visited when C references it again. This is correct behavior (ancestor chain, not global visited), but the spec never discusses shared references vs. cycles and whether the user will see confusing behavior when the same object appears expanded in multiple branches with different cycle detection results.

7. **AC7 is untestable as written.** "Given the depth-16 guard, when cycles exist deeper than the visited set can reach, then the depth guard still prevents runaway recursion." The visited set covers ALL ancestors -- there is no scenario where a cycle "exists deeper than the visited set can reach" because the visited set grows with depth. The only way to hit depth 16 without the visited set catching a cycle is a chain of 17+ unique objects with no cycles, which is not a cycle at all. This acceptance criterion describes an impossible scenario and provides no testable value.

8. **No consideration of `VariableValue::ObjectRef` at the root level.** The spec focuses entirely on `FieldValue::ObjectRef` cycles within `emit_object_children` / `build_object_items`. But the root entry point is in `flat_items` (line 498) and `build_items` (line 720), where `VariableValue::ObjectRef` is the starting point. The spec says "initial callers create the visited set with the root variable's object_id" (line 83 of spec), but the implementation tasks (Task 3, Task 4) say "create `let mut visited = HashSet::new()` and pass" -- an EMPTY set, not one pre-seeded with the root object_id. This is a contradiction. If the set starts empty, a self-referencing root object (field.id == root var's object_id) will only be caught by the `id == object_id` check inside the function, which happens to work, but the spec's own description of the design is inconsistent with the implementation instructions.

9. **No handling of `FieldValue::StringRef` that happens to have the same id as an ancestor object.** The spec says "StringRef is a leaf -- no recursion" (line 550 in code), so it's excluded from cycle detection. But what if a `StringRef` id collides with an ancestor `ObjectRef` id? The current code already skips StringRef for recursion, but the spec's visited set would contain the StringRef's id if it were ever checked. The spec doesn't acknowledge this edge case or explicitly state that StringRef ids live in a different namespace from ObjectRef ids in the hprof format. If they don't, a false positive cycle detection is possible on ObjectRef fields that happen to share an id with a previously-visited StringRef. The spec should state the assumption clearly.

10. **Test count claim is wrong.** The spec (Testing Strategy section) says "existing 369 tests must pass unchanged." This is a hardcoded number that will be stale by the time anyone implements this spec. If any other work lands first, the number changes and the statement becomes misleading. Use "all existing tests" instead of a specific count.

11. **The spec never addresses what happens when a user collapses an object that is part of a detected cycle.** When object A is expanded and shows B's `[cyclic]` marker back to A, what happens if the user collapses A? The cyclic marker under B should disappear, but does `collapse_object_recursive` (which uses `collect_descendants`) handle `OnCyclicNode` entries in the cursor correctly? The spec claims `collect_descendants` is "verified safe" (line 97) but doesn't explain how a cyclic marker -- which is NOT in `object_phases` or `object_fields` -- interacts with the collapse logic. If the cursor is on `OnCyclicNode` when its ancestor is collapsed, does the cursor reset work? Task 2 adds `OnCyclicNode` to the collapse reset pattern, but that pattern only fires when the entire frame is collapsed, not when a nested object is collapsed.

12. **The spec omits `selected_field_ref_id` and `selected_field_string_id` from the match-site inventory.** These functions (lines 226 and 313) match on `StackCursor` variants. Adding `OnCyclicNode` to the enum will cause compile errors in these functions too unless they have wildcard arms. The spec's Task 2 only lists three sites (`selected_frame_id`, `toggle_expand`, `flat_index`) but the compiler will flag more. Relying on "use the compiler to find every site" (Task 1 notes) is fine as a strategy, but then Task 2 should not pretend to be an exhaustive list -- it should say "including but not limited to" or just say "fix all compile errors" without enumerating a partial list that gives false confidence.

---

## Summary

The spec has the right general idea (ancestor-chain HashSet, terminal leaf markers, new cursor variant) but is sloppy in execution. Line number references are brittle and some are wrong. The task list is incomplete -- it misses at least one match site in `build_items` and doesn't account for all `StackCursor` match sites. AC7 describes an impossible scenario. The design description contradicts the implementation instructions on visited set initialization. Test coverage is insufficient: no navigation tests, no collapse-while-on-cyclic-node tests. The hardcoded test count is unprofessional. Fix these before handing this to a developer.
