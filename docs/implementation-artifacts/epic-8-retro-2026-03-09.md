## What Went Well

### Party Mode pre-epic planning eliminated false starts

The Algorithm Olympics, Red Team, and Pre-mortem sessions held
before the epic gave each story pre-validated technical decisions.
No hesitation, no false starts. Example: Story 8.1 had the
"sorted Vec vs suppress all_offsets" question already resolved
with data.

### Perfect story sequencing

8.0 (measurement tools) → 8.1 (main bottleneck) → 8.2
(allocation reduction) → 8.3 (parallelism). Each story built
cleanly on the previous one with zero friction.

### Zero regressions, zero Round 2 reviews

357 → 369 tests, none broken. No story required a Round 2
review — major improvement over Epic 3 (3/8 Round 2). Story
files with explicit anti-patterns ("Do NOT touch segment filter
logic") prevented common mistakes.

### Story files with excellent Dev Notes

Anti-patterns, architecture compliance, previous story
intelligence, and key code locations made each story
self-contained and implementation-ready.

### Refactoring work improved code quality

Post-story commits (ParsedRecord enum, named structs replacing
opaque tuples) left `first_pass.rs` more readable than before
the epic.

---

## What Didn't Go Well

### Target not fully reached within stories alone

The epic target was 30s → 15s (debug, no compiler optimizations).
Stories 8.0-8.3 alone achieved ~24s. Additional work via free
commits and quick specs was needed to reach ~20s. This is
inherent to optimization epics — fine-tuning is hard to plan
in stories — but the gap should be acknowledged.

### 5s freeze post-progress 100%

Documented in Story 8.2: `build_thread_cache` falls back to
`find_instance` (linear segment scan) when transitive IDs are
missing from `instance_offsets`. This creates a visible freeze
between progress bar completion and TUI display. Not resolved
during Epic 8.

### 4s unexplained gain

The drop from ~24s (after stories) to ~20s (after free commits)
is not precisely attributed. Likely caused by the ParsedRecord
enum refactor reducing allocations and improving branch
prediction, but not profiled.

### Cyclic reference freeze still unresolved

Action item #4 from Epic 3 retro — investigate the freeze when
expanding a variable on real dumps — was not addressed. This is
the second consecutive retro where this item is reported.

---

## Key Insights

1. **Optimization epics need a fine-tuning phase beyond
   stories** — the last 20% of gains come from polish work
   that is inherently hard to plan. This is normal as long as
   the work follows the same quality standards (tested, reviewed).

2. **Party Mode pre-epic planning is highly effective** —
   Algorithm Olympics, Red Team, and Pre-mortem sessions
   eliminated technical ambiguity before implementation started.
   Replicate for future epics.

3. **Benchmarks must target release mode** — the 4x difference
   between debug (~20s) and release (~4-5s) means debug
   benchmarks misrepresent real user experience. Release mode
   is the true metric.

4. **Action items must not be deferred more than once** — the
   cyclic reference freeze has been reported for two retros
   without action. New team agreement: no second deferral.
