## What Went Well

### TDD discipline held across all 6 stories

Red/Green/Refactor strictly followed. Test count grew from 54 (post-Epic 1) to 251. Zero panics
in production code, zero cross-story regressions.

### Payload-window pattern — key architectural discovery

Discovered during Story 2.3 review: without bounding each record's parse to a `payload_cursor`
slice, a malformed `header.length` could cause the parser to over-read into the next record.
The fix became the foundation for Stories 2.4 (tolerant recovery) and 2.5 (heap dump sub-record
parsing). A high-value architectural decision found at exactly the right time.

### `run_first_pass` infallible design

Making `run_first_pass` return `IndexResult` instead of `Result<_, HprofError>` (Story 2.4)
established a clean error model: all non-fatal parse failures become collected warnings.
This simplified all upstream code and made truncated-file handling graceful.

### Code reviews were substantive

Every story with notable complexity had a review that found real, non-cosmetic issues — all
fixed in the same session:
- 2.2: DoS vector via `Vec::with_capacity(num_frames)` (attacker-controlled allocation)
- 2.3: Cross-record over-read on malformed length metadata (2 High items)
- 2.4: Missing `Debug` derive, incorrect docstrings
- 2.6: Missing `{percent:.1}%` from progress bar template, trivially-true test assertion

### Objective reached

All 6 stories delivered. The indexing infrastructure is complete: sequential parse, precise
HashMap indexes, BinaryFuse8 segment filters, tolerant error handling, and a progress bar with
speed and ETA.

## What Didn't Go Well

### Security/architectural gaps discovered in review, not in dev

Pattern appeared in 3 stories:
- **2.2**: `Vec::with_capacity(num_frames as usize)` — NFR6 violation, caught in review
- **2.3**: Payload-window constraint missing — 2 High items, caught in review
- **2.6**: Progress bar missing percentage display — UX gap, caught in review

The review process is effective but is doing work that dev-time checklists should do first.

### UX of long initialization phases not covered in story spec

The progress bar from Story 2.6 covered the record-parsing pass correctly. But the
BinaryFuse8 filter construction phase (which happens after the main loop) had no feedback.
The user saw 100% on the progress bar, then silence while filters were built.

This was discovered at runtime — not in tests, not in review — and required multiple fix
commits:
```
feat: add ETA to segment filter progress bar
Fix: freeze scan bar during filter-build phase
Fix: lazy second bar + MultiProgress to prevent visual interference
Refactor: separate filter build phase into a second progress bar
Fix: show progress during heap dump inner loop and filter build phase
```

Root cause: the story spec did not model the indexing phase as two distinct user-visible
phases (parse records / build filters).

### Technical debt accumulating silently

Three items carried forward from Epic 2:
- `open_hprof_header` in `hprof-engine` — dead code since Story 2.6, still in public API
- `#[allow(dead_code)]` on `SegmentFilter::contains` — engine not yet built
- `#[allow(private_interfaces)]` on `HprofFile::segment_filters` — `SegmentFilter` not yet
  re-exported from `lib.rs`

All three will be resolved naturally in Story 3.1 when `hprof-engine` is built out.

---

## Key Insights

1. **Payload-window pattern is a cornerstone** — discovered in review, applied across 4 stories.
   Should be documented as a first-class architectural invariant in future story dev notes.

2. **Long init phases without feedback are a UX bug** — must be treated as an AC, not a
   post-delivery fix. If phase duration depends on file size, a visible feedback indicator is
   required by definition.

3. **TUI testing requires an explicit strategy** — `TestBackend` (ratatui) for render output,
   unit tests for model/state. Must be specified in ACs, not left implicit.
