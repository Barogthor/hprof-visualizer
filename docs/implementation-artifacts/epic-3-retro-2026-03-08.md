## What Went Well

### Async pattern (channel + poll) established in 3.4, reused in 3.6

The `pending_expansions` / `pending_strings` pattern using `thread::spawn` +
`mpsc::channel` + render-loop polling was designed once in Story 3.4 and replicated
in 3.6 for string loading with zero friction. Proves the architecture was sound.

### StatefulWidget pattern held across 6 stories

Established in 3.2 (`ThreadListState` + `SearchableList`), the `StatefulWidget`
pattern was reused in 3.3 (`StackState` + `StackView`) and extended through 3.8
without structural changes. Stable foundation.

### TDD discipline — 251 to 357 tests, zero regressions

Red/Green/Refactor strictly followed across all 8 stories. No cross-story
regressions detected. Zero panics in production code.

### Stories 3.7-3.8 made the tool actually usable

Real thread names from heap resolution, local variable class names, fixed frame
root correlation for jvisualvm dumps, and 2x performance improvement (60s → 30s
on 1 GB dump). Before 3.7, the tool was technically functional but practically
useless on real heap dumps.

### Dev notes quality

Every story file contains detailed resolution chains, format documentation, ASCII
diagrams, and implementation decisions. High-value context for future stories.

### `jvm_to_java()` module (3.3)

Clean, well-tested utility covering all JVM descriptor forms. Reused across
engine and TUI without modification.

## What Didn't Go Well

### Tool was unusable for 6 stories (3.1-3.6)

The tool showed `Thread-{serial}` placeholder names and had no local variables
visible on jvisualvm dumps (which lack `START_THREAD` records). This was not
discovered until Story 3.7 because:

1. **No manual validation on real dumps in ACs** — stories 3.1-3.6 had no AC
   requiring a test run against `heapdump-visualvm.hprof`
2. **Synthetic test builders masked real format behavior** — `HprofTestBuilder`
   always injects clean `START_THREAD` records, hiding the jvisualvm gap
3. **User feedback not integrated into process** — Florian noticed something
   was wrong as early as Story 3.2 but the signal was not captured

Root cause: the story prep process did not include observable validation criteria
on real assets.

### Frame root correlation bug lived silently from 3.3 to 3.7

The `GC_ROOT_JAVA_FRAME` correlation loop ran before thread synthesis in
`first_pass.rs`. On dumps without `START_THREAD`, the `threads` map was empty
during correlation, silently dropping all frame roots. Fixed in 3.7 by reordering
the synthesis block before the correlation loop. This bug affected 4 stories.

### 3/8 stories required Round 2 review fixes

- **3.2**: `list_threads()` called at 60 Hz in `render()` (H1); duplicate
  `search_query` state (H2); insufficient terminal cleanup (M1)
- **3.3**: Cursor not resetting on collapse (H1); GC root correlation O(n)
  instead of O(1) (H2)
- **3.6**: Warnings not propagated to StatusBar (H1); collapse not cleaning
  `pending_strings` (H2)

Pattern: integration bugs (state + render interaction) pass unit tests but fail
in review. Dev-time integration checks needed.

### Story 3.8 AC7 not met

Original target: <5s on 1 GB dump. Achieved: 8.4s (release), ~30s (debug/normal).
AC relaxed to <10s. Gap between release benchmarks and real user experience.

### 11 review follow-ups still open across 3.7 and 3.8

Notable items: manual e2e validation incomplete (3.7), ROOT_THREAD_OBJ truncation
warning missing (3.7), `all_offsets` memory scaling (3.8), AC7 performance gap
(3.8), incremental progress reporting (3.8).

## Key Insights

1. **Manual validation on real dumps is as critical as unit tests** — synthetic
   builders create a false sense of correctness. Every TUI story must include
   observable validation on real heap dumps.

2. **User feedback must be integrated into the process, not accidental** —
   Florian's observation at Story 3.2 went 5 stories before being addressed.
   The Project Lead should validate manually before marking stories as done.

3. **Synthetic test builders mask real hprof format behavior** — jvisualvm
   dumps without `START_THREAD` records are normal, not exceptional. Test
   assets must cover format variations.

4. **Solidify performance foundations before stacking features** — Epic 8
   (performance optimization) prioritized before Epics 4-7 to avoid building
   on a base that doesn't scale.

5. **Potential cyclic reference freeze** — Florian observed a freeze when
   expanding a variable on real dumps. Suspected infinite loop in recursive
   object resolution without cycle detection. Must investigate before Epic 8.

## Epic Reordering Decision

**Decision:** Epic 8 (First Pass Performance Optimization) is prioritized before
Epics 4-7.

**Rationale:**
- Review follow-ups from 3.8 point directly to problems Epic 8 solves
- `all_offsets` HashMap doesn't scale for 20 GB dumps
- Performance is the next user-facing irritant (tool is usable but slow)
- Building pagination (Epic 4) on top of unscalable indexing is risky

**Impact:** Sprint planning must be updated to reflect new epic order.
