## What Didn't Go Well

### 1. HIGH review findings in every single story — action item #2 from Epic 6 not applied

Every story had at least one HIGH/CRITICAL finding:
- 7.2: `FieldValue::Char` mapped to wrong theme role (AC3 violation)
- 7.2: toggle `+`/`-` styling marked done but not implemented
- 7.1: `SNAPSHOT_OBJECT_LIMIT` not enforced as hard cap during traversal
- 7.3: `Tab` no-op in search-active ThreadList and in Favorites

These are AC-by-AC gaps visible at spec read time. The pre-review semantic checklist
(Epic 6 action #2) was not applied.

### 2. Manual validation deferred to reviewer in 7.2

Story 7.2 AC5 (manual visual validation with `heapdump-visualvm.hprof`) was explicitly deferred
in the Dev Agent Record to the reviewer. Action item #1 from Epic 6 produced the right ACs but
did not ensure execution before review submission.

### 3. Story status / task checkbox inconsistency (recurring)

7.1 and 7.2 both had `Status: done` with `[ ]` unchecked tasks — flagged in reviews for the
second consecutive epic. Story lifecycle governance remains fragile.

### 4. Cross-story focus state machine bugs discovered late

`Focus::Favorites` (7.1) introduced two Tab handling gaps discovered only during 7.3 review.
The dependencies were documented but not cross-validated when writing 7.3.

## Action Items

### #1 — Dev agent executes manual validation AC before submitting to review

**What:** Any story with a manual validation AC (e.g., `cargo run -- assets/heapdump-visualvm.hprof`)
must be executed and confirmed by the dev agent — not deferred to the reviewer. Dev Agent Record
must include a line: "AC[N] manual validation: confirmed / deferred (reason)".

**Owner:** Amelia (Dev)
**When:** All Epic 9 stories
**Success criteria:** Zero "manual validation deferred to reviewer" in Dev Agent Records

### #2 — Pre-review cross-story dependency check for focus/input state machine

**What:** When a story touches `input.rs`, `app.rs` focus routing, or any `handle_*_input`
function, the dev agent must enumerate all `InputEvent` variants in every active `Focus` branch
and verify them against the help bar table before submitting. Use the existing help panel keymap
as a contract spec.

**Owner:** Amelia (Dev)
**When:** Stories 9.1, 9.6, 9.7, 9.8 (all touching input/focus)
**Success criteria:** Zero Tab/navigation no-op findings in Epic 9 reviews

### #3 — Story task checkboxes must be checked before setting `Status: done`

**What:** Dev agent must complete all `[ ]` → `[x]` updates in task list as part of the
story finalization step, before status is set to `done`.

**Owner:** Amelia (Dev)
**When:** Starting from Epic 9 Story 9.1
**Success criteria:** Zero "status done with unchecked tasks" findings in Epic 9 reviews

### #4 — Flag PinKey positional limitation in 9.6 story spec

**What:** Story 9.6 `g` navigate-to-object relies on `PinnedItem.key`. The positional `PinKey`
limitation (var_idx can drift on async reload) must be explicitly addressed or deferred in the
spec. SM to add a "Known Limitation: PinKey" section in the 9.6 story.

**Owner:** Bob (SM)
**When:** At 9.6 story creation
**Success criteria:** Story spec explicitly acknowledges the limitation and defines behavior
