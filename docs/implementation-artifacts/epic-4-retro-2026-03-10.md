## What Went Well

### Objective achieved — pagination works on 10K+ collections

Florian confirmed pagination works correctly with 10K entry
collections. Chunk sections (100/100/1000 layout), async page
loading, and keyboard navigation all function as designed.

### Zero regressions, zero Round 2 reviews

401 → 452 tests across the epic, none broken. No story required
Round 2 review — continuing the clean streak from Epic 8
(2 consecutive epics with 0 Round 2).

### Async pattern successfully reused

`pending_pages` / `poll_pages` pattern for async chunk loading
was directly modeled on `pending_expansions` / `poll_expansions`
from Epic 3. Zero friction reuse proves the architectural
pattern is sound and extensible.

### Clean Story 4.1 implementation

Collection pagination engine delivered with no debug issues.
Type dispatch per Java collection type (ArrayList, HashMap,
HashSet, LinkedList, ConcurrentHashMap, etc.) is cleanly
extensible. Cycle guards on chain walking prevent infinite loops.

### Manual validation by Project Lead generated real improvements

Florian's manual testing on real dumps after Story 4.2 review
led to 3 additional commits (inline values for String and boxed
primitives, auto-resolve strings, cleaner entry display). These
significantly improved the real user experience.

### Cyclic reference freeze finally resolved

Action item carried from Epic 3 → Epic 8 → resolved before
Epic 4 (commits `73d7d79` and `8e3ff3e`). No longer a concern.

## What Didn't Go Well

### AC10 missed in initial 4.2 implementation

ObjectRef entry values were hardcoded as leaf nodes ("Collection
entries are leaf nodes") despite AC10 explicitly requiring them
to be expandable. Required a substantial fix in code review:
new `OnCollectionEntryObjField` cursor variant, `find_entry()`
helper, `StartEntryObj`/`CollapseEntryObj` commands, and 3 new
tests.

### Extractors delivered without tests (4.1)

LinkedList and ConcurrentHashMap extractors were implemented
and marked complete but had zero test coverage. Caught in code
review (H1, H2). 6 tests added by reviewer.

### Manual validation still not formalized in ACs

Third consecutive retrospective flagging this issue. Florian's
manual testing happened informally after review, generating
3 commits outside the normal TDD cycle. The inline values code
initially contained UTF-8 byte-slice truncation panics (H1, H2
in post-review) — later fixed, but the bugs existed because the
code was added outside the red-green-refactor cycle.

### Code review reports not updated after fixes

The `claude-story-4.2-inline-values-post-review.md` report
still shows H1/H2 (UTF-8 panics) as open findings, but the
code has been corrected (uses `char_indices()` and
`chars().count()`). Review documents should reflect actual
state.

---

## Key Insights

1. **Manual validation generates real value but must be
   formalized** — Florian's testing led to inline values
   (major UX improvement), but the informal process produced
   code with bugs. Formalizing validation in ACs ensures the
   dev agent accounts for it during TDD.

2. **Extractors need test coverage at delivery** — implementing
   code without tests violates TDD principles. Code review
   should not be the safety net for missing tests.

3. **The async pattern is a proven reusable architecture** —
   three uses now (expansions, strings, pages) with zero
   structural issues.

4. **2 consecutive epics with zero Round 2 reviews** — story
   files with explicit Dev Notes, anti-patterns, and previous
   story intelligence are effective.
