## What Went Well

### TDD caught real bugs

The Red/Green/Refactor cycle caught the magic string length bug ("JAVA PROFILE 1.0.2" = 18 chars, not 19) during Story 1.2 test writing, before any integration code was written. Without TDD, this offset error would have silently propagated to header parsing.

### Code reviews were substantive

All three stories had code reviews that produced real, non-cosmetic improvements:
- Story 1.1: `Swatinem/rust-cache@v2` added, `--all-targets` on clippy, `println!` removed
- Story 1.2: `id_size` early validation added (fail-fast on invalid values)
- Story 1.3: `pub(crate)` visibility enforced, `id_size` validation in `parse_header`, cross-platform missing-path test, AC#7 integration test added

### Knowledge capitalized across stories

Completion notes were written with enough detail to be genuinely useful in subsequent story prep. The `MmapFailed` without `#[from]` decision (Story 1.2) was referenced and applied correctly in Story 1.3 without revisiting the architecture doc.

### Architecture boundaries held

The dependency constraint `hprof-cli` → `hprof-engine` → `hprof-parser` was respected. When FR1 end-to-end was implemented, the engine helper `open_hprof_header` was introduced to bridge the CLI without violating the boundary.

---

## What Didn't Go Well

### Scope creep via code review (2 out of 3 stories)

Story 1.3 had 4 review follow-ups, 2 rated High. The FR1 CLI end-to-end flow and the AC#7 integration test were not in the original story scope — they emerged from review. This means the story was shipped incomplete against implicit requirements, and the review acted as a second planning pass.

**Root cause:** Story prep did not systematically cover end-to-end flows, only component-level deliverables.

### Cross-platform blind spot at test-writing time

A Unix-specific path was used in the initial `mmap.rs` test (Story 1.3). This was caught by review, not by the developer. There was no systematic check for cross-platform correctness during development.

### Epic 2 prerequisites not formally tracked

`read_id()` utility, `tracing` crate, `xorf` (BinaryFuse8) — all mentioned in the architecture as required for Epic 2 — were never formally listed as prerequisites or pre-story tasks. They will need to be added to the workspace before or during Epic 2 Story 2.1.

---

## Key Insights

- The review process is effective but is currently doing some of the work that story prep should do.
- TDD is working. The process should be maintained and reinforced in Epic 2, which has more complex parsing logic.
- Documentation quality (completion notes, dev notes) is high and should be treated as a first-class deliverable.
